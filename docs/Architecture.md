## Архитектурное руководство

### Domain Layer (Доменный слой)

**Цель**: Чистая бизнес-логика, независимая от технических деталей

#### /abstracts

- **Что хранить**: Интерфейсы и типы бизнес-сущностей
- **Когда использовать**: Для описания структуры данных домена
- **НЕ класть**: API типы, DTO, UI модели, типизацию реквестов

#### /constants

- **Что хранить**: Бизнес-константы (лимиты, коэффициенты, правила)
- **Когда использовать**: Для неизменяемых бизнес-значений
- **НЕ класть**: UI константы, технические параметры

#### /enums

- **Что хранить**: Перечисления бизнес-значений (статусы, роли, типы)
- **Когда использовать**: Для ограниченных наборов бизнес-значений
- **НЕ класть**: Технические енумы (HTTP коды, UI состояния)

#### /services

- **Что хранить**: Доменные сервисы с чистой бизнес-логикой
- **Когда использовать**: Когда логика не принадлежит конкретной сущности
- **НЕ класть**: Сервисы с внешними зависимостями, API вызовы

### Infrastructure Layer (Инфраструктурный слой)

**Цель**: Взаимодействие с внешним миром (API, storage, внешние сервисы)

#### /repositories - осуществляет запросы к api и хранилищам посредством источников данных

- **Что хранить**: Классы для работы с API/storage
- **Когда использовать**: Для CRUD операций и запросов данных
- **НЕ класть**: Бизнес-логику, сложные вычисления

#### /datasources - предоставляет доступ к источнику данных с его личными настройками репозиторию

- **Что хранить**: Конфигурации и клиенты для внешних источников
- **Когда использовать**: Для инкапсуляции деталей подключения
- **НЕ класть**: Бизнес-логику, прямые api-вызовы

#### /resources (mappers) - обрабатывают ответы api приводя их к определенному формату, пригодному для работы в последующих слоях

- **Что хранить**: Преобразователи DTO ↔ Domain, обработчики ответов
- **Когда использовать**: Для трансформации данных между слоями
- **НЕ класть**: Бизнес-правила, валидацию

#### /dto - типы ответов api

- **Что хранить**: Типы API запросов/ответов
- **Когда использовать**: Для типизации внешних данных
- **НЕ класть**: Доменные типы, UI модели

#### /providers (глобальная)

- **Что хранить**: Фабрики и провайдеры инфраструктурных сервисов
- **Когда использовать**: Для предоставления repositories в application
- **НЕ класть**: Бизнес-логику, UI компоненты

### Application Layer (Слой приложения)

**Цель**: Оркестрация бизнес-процессов и сценариев использования

#### /services

- **Что хранить**: Сервисы с application логикой, resource services
- **Когда использовать**: Для простых операций и проксирования
- **НЕ класть**: Сложные многошаговые сценарии, UI логику

#### /use-cases

- **Что хранить**: Сложные бизнес-сценарии
- **Когда использовать**: Для многошаговых операций с побочными эффектами
- **НЕ класть**: Простые CRUD операции, UI логику

#### /providers (глобальная)

- **Что хранить**: Фабрики сервисов и use cases
- **Когда использовать**: Для предоставления application сервисов в presentation
- **НЕ класть**: Инфраструктурные детали, UI компоненты

### Presentation Layer (Презентационный слой)

**Цель**: UI логика, состояние интерфейса, взаимодействие с пользователем

#### /stores

- **Что хранить**: Svelte stores для управления состоянием
- **Когда использовать**: Для реактивного состояния UI
- **НЕ класть**: Бизнес-логику, прямые API вызовы

#### /view-models

- **Что хранить**: Модели представления с UI логикой
- **Когда использовать**: Для сложных форм и компонентов
- **НЕ класть**: Бизнес-правила, прямую работу с API

#### /schemas

- **Что хранить**: Схемы валидации форм (zod, yup)
- **Когда использовать**: Для клиентской валидации
- **НЕ класть**: Бизнес-валидацию, серверные правила

#### /constants

- **Что хранить**: UI константы (размеры, цвета, конфигурации)
- **Когда использовать**: Для настроек отображения
- **НЕ класть**: Бизнес-константы, API параметры

### Правила зависимостей

1. **Domain** → не зависит ни от кого
2. **Infrastructure** → использует Domain
3. **Application** → использует Domain и Infrastructure (через интерфейсы)
4. **Presentation** → использует Application (через провайдеры)

### Ключевые принципы

- **Однонаправленные зависимости**: Внешние слои зависят от внутренних
- **Изоляция по сущностям**: Каждая сущность имеет свою структуру в каждом слое
- **Провайдеры как мосты**: Связывают слои без прямых зависимостей
- **Shared для общего**: Используется для переиспользуемых компонентов внутри слоя
